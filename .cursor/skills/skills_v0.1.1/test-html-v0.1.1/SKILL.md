# HTML 验收

本 skill 是 `../html-authoring-v0.1.1/SKILL.md` 和 `../courseware-generator-v0.1.1/SKILL.md` 的交付闸门。它不生成页面，只判断页面是否可以交付。验收失败时，应回到对应生成 skill 修复，再重新验收。

## 被测产物

单页和多页都以本地目录为准：

```text
pages/<slug>/index.html
```

多页课件还必须存在：

```text
pages/<slug>/courseware-shell.js
```

## 工具边界

验收只使用本地可用方式：浏览器人工检查，或本地 Playwright 打开本地 URL。不要依赖云端平台验收工具、平台文件标识或外部资源 ID。优先通过本地 HTTP 服务打开页面，避免 `file://` 下脚本、字体、iframe 或下载能力表现不一致。

```bash
cd pages/<slug> && python3 -m http.server 8765
```

默认 URL：

```text
http://127.0.0.1:8765/index.html
```

## 验收输入

从以下位置抽取 must-cover：

- 用户原始需求；
- `index.html` 中的 `<!-- spec: ... -->`；
- `assets-manifest.md`；
- 多页课件的 `outline.md`。

must-cover 至少包含四类：`core-loop`、`require`、`forbid`、`delivery`。多页课件额外包含 `shell`、`pages`、`navigation`、`state`、`quiz-score`、`scorm-package`。

抽取 must-cover 时不能只看页面里的显式 spec，也不能只看用户最后一句。要从全部用户消息中识别四类信号：交互动词和对象，例如“拖到数轴”“点击生成”“提交答案”；声明式约束，例如“不要滚动”“A4 打印”“10 道题”“适合小学生”；核心体验闭环，例如游戏是否能从开始走到反馈和重置；交付可达性，例如本地 URL 是否能打开、素材是否真实、入口是否可用。

`forbid` 项按全局约束处理，不是抽样检查。用户说不能出现某种内容、不能溢出、不能滚动、不能使用弹窗、不能用虚构素材时，应检查页面全局、关键视口和相关代码路径。`require` 项也不能只用元素存在代替，需要确认它在实际操作中发挥作用。

## 硬门槛

以下任一项失败，都不能声明交付：本地页面无法打开；主内容不可见；单页产物出现横向溢出或核心操作不可达；核心闭环无法走完；用户明确禁止项出现；关键按钮或入口无反馈；多页课件壳未加载、页数不匹配、无法翻页或任一 page-data 产生横向/纵向滚动；练习页要求成绩但无法产生可捕获成绩消息。

核心闭环的判定标准是“用户能完成任务并看到结果”。对于游戏，要能开始、操作、判定、反馈、重置或进入下一轮；对于测验，要能作答、确认、看到对错/解析、继续或重做；对于拖拽/分类，要能真实改变归属或位置、检查结果并反馈；对于模拟器，要能控制运行状态并看到状态变化。

## 人工验收流程

1. 启动本地 HTTP 服务并打开页面。
2. 先确认页面可打开、无明显报错、主内容可见。
3. 按 must-cover 逐项操作，不只看元素是否存在。
4. 记录每一项通过或失败；失败时写清楚页面现象和应回修的位置。
5. 输出验证结论卡。

单页重点看：核心互动是否走完、按钮是否有反馈、答题/拖拽/模拟器是否能收尾、素材是否真实、是否有横向溢出、滚动是否影响核心任务。多页重点看：壳是否加载、缩略图是否出现、点击和键盘翻页是否可用、页数是否匹配 outline、每个 page-data 是否在 960×540 内无滚动、互动状态是否恢复、练习页是否发送 `cwScore`。

人工验收时要主动寻找“看起来完整但不能用”的问题：按钮只有 hover 没有点击效果，拖拽只移动视觉不改变答案状态，答题只显示分数不显示解析，模拟器只有背景动画没有数据状态，打印页屏幕可读但打印预览断裂。发现这类问题时，结论应记为未通过或部分通过，而不是“体验可优化”。

## 本地 Playwright 验收

需要自动化时读取 `references/test-templates.md`。脚本应打开本地 URL，并覆盖真实行为，而不是只断言元素存在。

最低检查项：

```text
□ body 或主容器可见
□ 视口宽度 375 / 768 / 1280 下无横向溢出，核心操作可达
□ core-loop 端到端触发，并出现反馈或状态变化
□ forbid 项全局不存在
□ 每个关键按钮点击后有可见反馈
□ 外部图片和音频能加载，或 assets-manifest 声明自绘
```

多页课件额外检查：

```text
□ .cw-root 和 .cw-main-iframe 可见
□ 缩略图 iframe 数量大于 0
□ template.page-data 数量与 outline 页数一致
□ 每个 page-data iframe 内 scrollWidth <= clientWidth 且 scrollHeight <= clientHeight
□ 翻页后当前页变化
□ 互动页离开再返回后状态恢复
□ 有练习页时捕获到 cwScore message
□ 如果有 SCORM 包按钮，点击后没有页面级错误
```

断言应优先检查状态变化、可见文本、DOM 归属、计数、消息、视口尺寸和错误日志。存在性断言只适合做入口检查，不能替代核心行为检查。对“不能出现”的要求使用全局文本、类名、样式或截图特征检查；对“数量必须一致”的要求直接计数；单页产物至少覆盖移动、平板和桌面三个视口，检查横向溢出和核心操作可达性。多页课件要进入主预览 iframe或逐页 iframe 检查每一页的 `scrollWidth <= clientWidth`、`scrollHeight <= clientHeight`，而不是只检查外层壳。

## 品类补盲

验收脚本或人工清单应按品类补齐常漏项。游戏和闯关类要检查新题、重置、得分和失败反馈；测验类要检查确认后反馈、正确答案/解析、下一题、结果页和重做；拖拽/分类类要检查真实移动、拒绝错误目标、提交检查和复位；动画/模拟类要检查开始、暂停、重置和实时状态；课件类要检查 960×540 页面不溢出不滚动、缩略图与主预览一致、翻页后互动状态可恢复。

## 失败处理

默认先判断页面实现是否缺失。只有选择器错误、脚本语法错误、或诊断确认页面正确时，才修改测试脚本。不要为了通过验收删除 must-cover，也不要把核心闭环降级成“按钮存在”。连续失败时先打印 DOM 状态、事件结果、控制台错误或手动复现，再决定修页面还是修测试。

同一失败重复出现时，不能原样重跑并给出新的乐观结论。必须增加诊断证据，例如点击前后文本、类名、数据状态、iframe 页码、消息捕获记录或截图差异。基础设施失败要如实写入结论卡，不能把未运行的检查算作通过。

## 结论卡

```text
## 验证结论
- 状态：完全通过 / 部分通过 / 未通过
- 需求覆盖：N/M 项
- 未覆盖项：无 / <列出原因>
- 下一步：交付 / 修复后复测
```

只有本地打开、core-loop、forbid、多页壳加载这些硬门槛通过后，才可以声明交付。
