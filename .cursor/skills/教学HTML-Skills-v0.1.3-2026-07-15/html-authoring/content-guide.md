# 单页 HTML 内容指南

本文件用于非数学或学科不明的单页教学 HTML。数学任务不要读本文件里的视觉建议作为最终权威，改读 `math-design/workflow.md`。

## 适用场景

- 教学动画、互动练习、小游戏、教学海报；
- 可打印练习册、字帖、试卷等 HTML 打印材料；
- 修改已有单页 HTML。

不适用于多页翻页课件。

## 内容组织

先判断信息量，再决定骨架：

| 信息量 | 骨架 |
|---|---|
| 单一主题、单屏可容纳 | 单页布局 |
| 3-5 个并列模块 | Tabs |
| 6 个以上条目或大量列表 | Tabs + Accordion |
| 打印材料 | A4 `<section class="page">` 分页 |

不要把大量内容直接堆成长滚动页。用户提到“练习 + 答案”“多课时”“不同模式”“N 道题”时，优先用 Tabs 或 Accordion 管理。

## 内容源契约

生成前先从用户需求中提取内容源：学科、年级、课题、学习目标、必须出现的原文/公式/定义/数据/题目、素材、禁止项。用户明确给出的文本、公式、题目、选项、答案、案例和数据必须原样保真，不能为了排版顺手改写；没有给出具体内容时，允许生成通用教学内容，但要与学段和题目匹配，不能编造“来自教材”的权威来源。

每个页面或模块都要有明确教学任务。最低结构是：主题句说明这部分学什么，主体内容给出定义/步骤/例子/图示/问题，操作或反馈告诉学生下一步做什么。不要只做漂亮版式骨架，也不要只写“点击开始学习”“这里展示知识点”这类空话。

## 最低内容标准

按产物类型检查内容是否充实：

| 类型 | 最低内容要求 |
|---|---|
| 概念讲解 | 定义或规则 + 2-4 个要点 + 至少 1 个例子或反例 |
| 过程演示 | 起点状态 + 步骤序列 + 每步变化 + 结论 |
| 互动练习 | 题干 + 可操作输入/选项 + 判定规则 + 反馈/解析 + 重做或下一题 |
| 小游戏/闯关 | 目标 + 规则 + 得分/进度 + 成功/失败反馈 + 新一轮或重置 |
| 拖拽/分类/排序 | 对象集合 + 目标区域/顺序规则 + 正误判定 + 错误提示 + 复位 |
| 模拟器/实验 | 可控变量 + 状态显示 + 观察结果 + 暂停/重置 + 解释提示 |
| 海报/知识卡 | 核心标题 + 3-5 个信息块 + 关键例子/图示 + 结论或行动提示 |
| 打印材料 | 题区 + 作答空间 + 页眉/标题 + 答案或解析区，按需要拆成 A4 分页 |

不允许出现“标题 + 一句笼统话”的空壳模块。讲解内容不能只有抽象口号，练习不能只有选项没有答案，互动不能只有按钮没有反馈，演示不能只有动画没有状态含义。

## 信息密度与滚动

单页内容密度必须主动控制。互动练习、小游戏、模拟器和演示台应优先把当前任务需要的内容放在首屏，辅助说明压缩为短句、chip、提示卡或可切换状态。长文档、阅读材料、资料汇编、打印预览和长表格可以滚动，但要有清楚的标题层级、分区、目录或锚点，避免读者迷失。

内容超过一屏时，优先判断是否应该改成 Tabs、Accordion、步骤分页、题目分页或拆到 `courseware-generator` 多页课件。Tabs 用于并列主题，Accordion 用于同类条目，步骤分页用于有顺序的过程，题目分页用于多题练习。每一种切换都要有当前状态提示，例如“第 2/5 题”“步骤 3：观察变化”“模式：挑战”。

## 学段与表达

小学内容要使用短句、具体对象和可观察反馈，少用抽象术语堆叠；初高中内容可以增加公式、推导和概念对比，但必须保留例子或应用场景。语文、英语类内容要保留原文或语料的完整性；数学、科学类内容要保证公式、单位、数值、图示关系一致；历史地理类内容要避免把人物、年代、地点和因果关系写混。

如果用户只给出宽泛主题，例如“做一个光合作用小游戏”，应补齐基础教学内容：学生任务、核心概念、操作规则、反馈方式和总结句。不要把宽泛主题直接变成视觉装饰页。

## 交互规则

- 用户提到的每个具体数值、模式、分类、难度，都要在 UI 上有对应控件。
- 每个 `<button>`、`[role="button"]`、`.btn`、可点击卡片都必须绑定事件。
- 模拟器或实时动画必须有开始、暂停、重置和状态显示。
- 答题、拖拽、匹配、分类、排序等判定类任务必须有提交或检查答案、重置、结果反馈。
- UI 文案承诺什么操作，页面就必须真的支持该操作；做不到时改文案。
- 交互反馈要解释“为什么”，不能只显示“正确/错误”。至少提供正确答案、关键依据、下一步建议中的一种。
- 多题练习要有题号、进度、得分或完成状态；不能把多道题一次性堆在同一屏。

## 题目与答案

题目必须可判定。选择题要有明确正确项和错误项，填空题要有可接受答案，拖拽/分类要有目标映射，排序题要有正确顺序。答案和解析必须与题干一致，不能出现选项数量和判定逻辑不匹配。

生成练习时优先使用用户给定题目；用户未给题时，题目难度应与年级和主题匹配，并覆盖核心目标。不要为了凑数量生成重复题、无意义题或和主题无关的题。

## 打印材料

触发词：打印、A4、练习册、字帖、习题册、试卷、N 页。

```css
@page { size: A4 portrait; margin: 15mm; }
.page { page-break-after: always; width: 190mm; min-height: 277mm; }
@media print { .no-print, .print-btn, header, footer { display: none; } }
```

打印页必须包含真实内容，不能留下 `{{placeholder}}`、`${data.xxx}`、`TODO` 或运行时注入占位。

## 素材与自包含

- 外部图片、音频、字体必须来自真实可访问的稳定 HTTPS URL，并记录在 `artifact-spec.assets`，且只能作为非核心增强。
- 远程资源必须提供 fallback；禁止虚构路径、占位 URL、本机/相对运行路径和大体积 base64 音视频。
- 不需要真实媒体时，用 SVG、Canvas 或 CSS 自绘。
- HTML 应能独立运行；所有数据以字面量写入 HTML/JS。
- **需要 AI 配图时**：先读本 skill 的 `references/image-generation.md`，命中式增强后调用 `generate_images`；工具不可用则自绘并声明。数学示意图优先自绘，不强制套语文/历史风格库。

## 内容自检

```text
□ 单个 <slug>.html 已准备为完整最终内容
□ <head> 内含可解析的 artifact-spec JSON
□ 学科、年级、主题、学习目标已体现在内容中
□ 用户给定原文/公式/数据/题目没有被篡改
□ 每个模块都有教学任务，不是空版式
□ 讲解有要点和例子，练习有答案和反馈
□ 内容超量时已选择合适组织方式，滚动不会影响核心任务完成
□ core-loop 每步可触发
□ require 项全部出现
□ forbid 项未出现
□ 每个按钮和可点击元素都有反馈
□ 素材来自 artifact-spec.assets 或已声明自绘
□ 没有同目录、本机或 Skill 内部运行依赖
```
